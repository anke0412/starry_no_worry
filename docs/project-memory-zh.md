# Starry No Worry 项目记忆

这份文档用于沉淀当前项目中长期稳定、值得持续遵守的目标、术语、架构方向和协作约定。  
它不是变更日志，也不追踪每一次分支细节；重点是帮助后续开发、调研、代码生成和代码审阅保持一致。

## 1. 项目目标

- 项目名：`starry_no_worry`
- 长期目标：做成全功能占星排盘平台
- 当前开发方式：按阶段递增推进
- 第一阶段目标：可上线的占星工具型 MVP
- 第二阶段重点：
  - 更专业的计算能力
  - 更强的盘型体系
  - 后续 AI 解读能力
- 长期记忆优先级：
  - 优先记住 `plan`、项目目标、术语体系、架构方向
  - 不需要过度记忆每一次分支、零散提交和短期操作细节

## 5. 当前已确认的生成框架方向

为了后续扩展更多盘型，后端已经开始转向统一的盘型生成框架。核心四类生成器方向是：

- 单体衍生：
  - 本命 → 流年 / 次限 / 日返 / 月返
- 双体比较：
  - A + B → 比较盘
- 双体衍生：
  - A + B + 一个目标时点 → 关系流年盘
- 双体融合：
  - A + B → 组合盘 / Davison / 马盘

已完成的重要架构动作：

- 已建立共享 chart generation framework
- 已明确的复用方向包括：
  - `transit` 作为单体衍生模式复用统一生成流程
  - `synastry` 作为双体比较模式复用统一生成流程
  - 日返盘建立在共享生成框架之上
- 高级盘型的实际 rollout 状态以当前 active Phase 2 plan 和 `work/decisions/` 为准，不要仅凭旧 spec、旧测试或旧记忆判断某个声明盘型已经源码完整。

后续新增高级盘型时，应优先复用这套生成框架，而不是重新各自拼接独立 orchestration。

## 6. 代码与产品注意事项

- 用户非常在意中文术语、占星专业表达和视觉细节
- 人名展示必须严格使用用户输入，不要乱追加后缀
- 行星名、星座名都需要中文
- 相位显示、tooltip、宫位 / overlay 表述需要尽量精确到“谁和谁”
- 表格结构和轮盘结构都要保持专业可读
- 第二阶段稳定公共能力已经覆盖：可选点位、多宫制、相位集合 / 容许度、统计面板
- 更细的盘型完成度与交互缺口应回看当前 active plan / `work/decisions/`，避免把短期审计状态误当成长久记忆
- 在前端测试、tooltip、可视化修正时，要默认用户会检查很细的小问题
- backend 测试环境以 `.venv312` 为准
- backend 测试基线不要写死固定 `passed` 数：
  - 当前可复核基线是：在 `backend` 目录执行 `../.venv312/bin/python -m pytest --collect-only -q`
  - 当前仓库状态可收集 `100 tests`
  - 以后优先维护“如何复核”和“当前收集规模”，不要只记容易过期的通过数

## 7. 协作运行约定

- 主 agent 负责先读取这两份项目记忆，以及当前任务对应的 task / plan 文档，再决定下一步任务
- 主 agent 负责把工作拆给三类子 agent：
  - `implement`
  - `review`
  - `verify`
- 主 agent 负责汇总子 agent 结果，并回写当前 plan 的勾选状态
- 主 agent 负责持续维护测试基线表述，避免记忆中留下容易过期的固定通过数
- 主 agent 负责沉淀审阅结论，明确哪些结论已经确认、哪些仍是待验证风险

## 8. 子代理协作约定

以后在使用子代理时，尤其是以下场景：

- 代码生成
- 调研
- 代码审阅
- 验证
- 调试

子代理名字需要带上自己的功能目标，方便人工审阅，不要只用模糊名字。

推荐命名风格：

- `impl_transit_generator`
- `review_synastry_adapter`
- `research_composite_chart_rules`
- `verify_backend_chart_suite`
- `debug_pytest_import_path`

原则：

- 名字里直接体现任务目标
- 让人一眼看出它是实现、调研、审阅还是验证
- 优先体现当前模块，而不是抽象昵称

## 9. 文档使用原则

这份文档用于：

- 后续继续开发时快速恢复上下文
- 对齐术语和产品目标
- 约束子代理协作方式
- 减少因为上下文变长导致的方向漂移
